## 0. CONTEXTO DO PROJECTO (não alterar nada disto, só ler)

O Xclusive é uma plataforma de conteúdo por subscrição/gorjetas (estilo OnlyFans) já em produção com:

**Stack:**
- Frontend utilizador: React + TypeScript
- Backend: Express + Drizzle ORM
- Base de dados: Postgres (via Drizzle)
- Monorepo com pasta `artifacts/`

**Já existe e funciona (NÃO MEXER):**
- 14 páginas frontend: landing, registo (2 passos), onboarding, feed, explorar, reels, mensagens, notificações, perfil, carteira, KYC (6 passos com câmara), monetização, definições
- +35 endpoints API: auth, users, posts, stories, reels, comments, tips, subscriptions, creator plans, messages, notifications
- Tabelas Drizzle existentes: `users`, `posts`, `post_media`, `likes`, `comments`, `saved_posts`, `subscription_plans`, `subscriptions`, `purchases`, `stories`, `conversations`, `conversation_participants`, `messages`
- Sistema de monetização: gorjetas (com guard anti-self-tip), PPV por post, planos de subscrição por criador, carteira interna (saldo + ganhos separados)
- Modo mock: todas as páginas do utilizador funcionam sem DB usando dados fictícios

**Objectivo desta tarefa:** construir um painel de administração NOVO, isolado, sem alterar nenhuma rota, página ou comportamento já existente da app do utilizador.

## 2. ALTERAÇÕES AO SCHEMA (migrations Drizzle)

### 2.1 Adicionar coluna `role` a `users`

```sql
ALTER TABLE users ADD COLUMN role VARCHAR(20) NOT NULL DEFAULT 'user';
-- valores possíveis: 'user', 'creator', 'admin', 'superadmin'
```

### 2.2 Nova tabela `reports`

```sql
CREATE TABLE reports (
  id SERIAL PRIMARY KEY,
  reporter_id INTEGER REFERENCES users(id),
  target_type VARCHAR(20) NOT NULL, -- 'post', 'comment', 'user', 'message'
  target_id INTEGER NOT NULL,
  reason VARCHAR(50) NOT NULL, -- 'nudity_minor', 'spam', 'harassment', 'copyright', 'other'
  description TEXT,
  status VARCHAR(20) NOT NULL DEFAULT 'pending', -- 'pending', 'reviewing', 'resolved', 'dismissed'
  resolved_by INTEGER REFERENCES users(id),
  resolved_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW()
);
```

### 2.3 Nova tabela `withdrawal_requests`

```sql
CREATE TABLE withdrawal_requests (
  id SERIAL PRIMARY KEY,
  creator_id INTEGER REFERENCES users(id) NOT NULL,
  amount NUMERIC(12,2) NOT NULL,
  method VARCHAR(30) NOT NULL, -- 'bank_transfer', 'multicaixa_express', etc.
  destination_details JSONB,
  status VARCHAR(20) NOT NULL DEFAULT 'pending', -- 'pending', 'approved', 'rejected', 'paid'
  processed_by INTEGER REFERENCES users(id),
  processed_at TIMESTAMP,
  notes TEXT,
  created_at TIMESTAMP DEFAULT NOW()
);
```

### 2.4 Nova tabela `audit_log`

```sql
CREATE TABLE audit_log (
  id SERIAL PRIMARY KEY,
  admin_id INTEGER REFERENCES users(id) NOT NULL,
  action VARCHAR(60) NOT NULL, -- 'user_suspend', 'user_delete', 'withdrawal_approve', etc.
  target_type VARCHAR(30),
  target_id INTEGER,
  details JSONB,
  ip_address VARCHAR(45),
  created_at TIMESTAMP DEFAULT NOW()
);
```

### 2.5 Nova tabela `platform_settings`

```sql
CREATE TABLE platform_settings (
  key VARCHAR(60) PRIMARY KEY,
  value JSONB NOT NULL,
  updated_by INTEGER REFERENCES users(id),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- seed inicial:
-- 'commission_rate' -> {"value": 20} (percentagem retida pela plataforma)
-- 'maintenance_mode' -> {"enabled": false}
-- 'allowed_countries' -> {"list": ["AO", "PT", "BR", ...]}
-- 'min_withdrawal_amount' -> {"value": 5000}
```

## 3. AUTENTICAÇÃO ADMIN

- Criar seed que promove um utilizador existente a `role='admin'` via script `seed-admin.ts` (recebe email por argumento).
- Rota `POST /api/admin/auth/login` — mesmo mecanismo de sessão/JWT já usado na app, mas valida `role IN ('admin','superadmin')` antes de emitir o token.
- Middleware `requireAdmin.ts`: rejeita com 403 qualquer request a `/api/admin/*` cujo utilizador não tenha `role` admin/superadmin.
- `AdminAuthContext.tsx`: mesmo padrão de Context API que a app principal já usa para auth, mas separado (sessão/token diferente da app do utilizador, para não misturar sessões).
- TTL de sessão admin mais curto que o da app normal (ex: 4h) por segurança.

## 4. DASHBOARD PRINCIPAL (`Dashboard.tsx`)

### 8 KPIs no topo (cards)
1. Total de utilizadores — `SELECT COUNT(*) FROM users`
2. Total de criadores — `SELECT COUNT(*) FROM users WHERE role='creator'`
3. Novos registos hoje — `SELECT COUNT(*) FROM users WHERE created_at >= CURRENT_DATE`
4. Receita total (gorjetas + subscrições + PPV) — soma de `purchases` + `subscriptions` no período
5. Comissão da plataforma no mês — receita total × `commission_rate`
6. Posts publicados hoje — `SELECT COUNT(*) FROM posts WHERE created_at >= CURRENT_DATE`
7. Denúncias pendentes — `SELECT COUNT(*) FROM reports WHERE status='pending'`
8. Levantamentos pendentes — `SELECT COUNT(*) FROM withdrawal_requests WHERE status='pending'`

### 4 Gráficos (Recharts)
1. **Crescimento de utilizadores** — LineChart, eixo X = últimos 30 dias, eixo Y = novos registos/dia
2. **Receita por dia** — BarChart, últimos 30 dias, empilhado por tipo (gorjetas / subscrições / PPV)
3. **Top 10 criadores por ganhos** — BarChart horizontal, período seleccionável (7/30/90 dias)
4. **Distribuição de utilizadores por país** — PieChart ou BarChart

### Feed de actividade em tempo real
Lista das últimas 20 acções relevantes (novo registo, nova denúncia, novo levantamento pedido, KYC submetido) — polling a cada 15s ou websocket se já existir infraestrutura.

## 5. GESTÃO DE UTILIZADORES (`Users.tsx` + `UserDetail.tsx`)

**Tabela principal:**
- Colunas: avatar, nome, username, email, role, país, estado (activo/suspenso), data de registo
- Filtros: por role, por país, por estado, por data de registo, pesquisa por nome/email/username
- Paginação server-side

**Página de detalhe com 4 abas:**
1. **Perfil** — dados pessoais, editar inline (nome, bio, país)
2. **Actividade** — posts, comentários, histórico de login
3. **Financeiro** — saldo, ganhos, histórico de transacções, subscrições activas
4. **Moderação** — denúncias recebidas, denúncias feitas, histórico de suspensões

**Acções:**
- Suspender/reactivar conta (`PATCH /api/admin/users/:id/status`)
- Eliminar conta (soft delete, `DELETE /api/admin/users/:id`)
- Editar dados inline (`PATCH /api/admin/users/:id`)
- Alterar role (`PATCH /api/admin/users/:id/role`)

Todas as acções destrutivas/sensíveis gravam entrada em `audit_log`.
